# Multi-hop, Load Balance и приоритеты групп

Версия 3.2.1 поддерживает advanced targets: группа может быть не только списком запасных серверов, но и отдельной целью подключения.

## Режимы группы

`Failover` - классический режим. Клиент выбирает один сервер из группы и запускает его как обычный профиль. Порядок списка используется стратегией `По порядку`.

`Multi-hop` - цепочка серверов. Первый сервер является первым hop, последний сервер является exit. В sing-box это собирается как несколько outbound с `detour`, а итоговый outbound сохраняет tag `proxy`, чтобы существующая маршрутизация работала без изменений.

`Load Balance` - группа sing-box `urltest`. Каждый сервер получает отдельный outbound, а tag `proxy` становится group outbound, который выбирает рабочий/быстрый сервер по URL-test.

## Как создать

1. Откройте вкладку `Серверы`.
2. Выберите сервер и нажмите `Failover`, чтобы создать базовую группу из соседних серверов.
3. Выберите созданную группу в таблице Smart Connect / Failover.
4. Нажмите `Редактировать`.
5. Выберите режим: `Failover`, `Multi-hop` или `Load Balance`.
6. Добавьте или удалите серверы.
7. Кнопками `Выше` и `Ниже` задайте ручной приоритет.
8. Сохраните группу.

## Запуск группы

Выберите группу в таблице и нажмите `Запустить группу`.

Для `Failover` будет запущен один сервер по стратегии группы. Для `Multi-hop` и `Load Balance` будет собран отдельный sing-box config с несколькими outbounds.

## Multi-hop порядок

Порядок важен:

```text
1. Entry hop
2. Middle hop
3. Exit hop
```

В runtime config это выглядит концептуально так:

```json
[
  {"tag": "hop-1", "type": "vless"},
  {"tag": "hop-2", "type": "trojan", "detour": "hop-1"},
  {"tag": "proxy", "type": "vless", "detour": "hop-2"}
]
```

Маршрутизация по-прежнему отправляет VPN-трафик в `proxy`, поэтому правила split tunneling менять не нужно.

## Load Balance параметры

`LB interval` - как часто sing-box проверяет кандидатов.

`LB tolerance, ms` - насколько близкими считаются задержки серверов. Чем меньше значение, тем агрессивнее выбирается самый быстрый сервер.

Клиент использует `urltest` outbound. Это практичный режим автоматического выбора лучшего рабочего сервера внутри группы. Точный per-outbound трафик sing-box в текущем UI не отдаёт, поэтому usage для серверов Load Balance распределяется между участниками группы поровну.

## Долгосрочная статистика

Для сервера сохраняется:

- количество запусков;
- общее время подключения;
- суммарный download/upload;
- последняя дата подключения и отключения;
- история latency/success/failure.

Для advanced-группы также сохраняются общие usage-счётчики. Статистика хранится локально в папке данных приложения.

## Ограничения

- Multi-hop требует минимум два сервера.
- Load Balance требует минимум два сервера.
- Если один hop в Multi-hop недоступен, вся цепочка не подключится.
- Для точной per-server статистики внутри Load Balance нужен отдельный сбор per-outbound counters; сейчас используется равномерное распределение session traffic.
- Health monitor для advanced-группы перезапускает всю группу, а не отдельный сервер.
